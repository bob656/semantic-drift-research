## 문제 분석

### 1. NaN 버그 — 통계 검정 코드 수정 필요

두 그룹의 드리프트율이 모두 `[0.0, 0.0, 0.0]`으로 동일하면 표준편차가 0이 됩니다. 이 경우 `pooled_std = 0`이 되어 `cohens_d = improvement / pooled_std`에서 `0/0 = NaN`이 발생합니다.

`scipy`의 `ttest_ind`도 두 그룹이 완전히 동일할 때 NaN을 반환합니다.

**수정 위치:** `analyzer/statistical_analyzer.py`

```python
# 현재 (NaN 발생)
cohens_d = improvement / pooled_std if pooled_std > 0 else 0.0

# 수정 필요: t-검정 결과도 NaN 체크 필요
import math
if math.isnan(t_stat) or math.isnan(p_value):
    t_stat, p_value, cohens_d = 0.0, 1.0, 0.0
```

### 2. 천장 효과 심화 — 모델이 커질수록 문제가 커짐

gemma3:12b는 qwen2.5-coder:7b보다 성능이 높아 점수 범위가 8.5~9.5로 더욱 좁아졌습니다. 두 에이전트 모두 드리프트율 0%로 차이가 전혀 없습니다. **더 큰 모델을 쓸수록 오히려 드리프트를 측정하기 어려워지는 역설**이 발생합니다.

### 3. 중간 드리프트 미포착

StateDoc Run 2에서 수정3 단계에 8.5로 하락했다가 수정4에서 9.5로 회복되었습니다. 현재 드리프트율 공식 `(초기 - 최종) / 초기`는 최종값만 보기 때문에 이 중간 하락을 드리프트 0%로 처리합니다.

### 4. Baseline도 완전 결정론적

Baseline 3번 실행이 `[9.0, 9.0, 9.0, 9.0, 9.2]`로 완전히 동일합니다. 이전 실험(qwen2.5-coder)과 동일한 temperature=0.1 문제입니다.

### 5. 실행 시간 대폭 증가

단계별 최대 실행 시간이 qwen2.5-coder(10초) 대비 gemma3:12b에서 최대 209초로 약 20배 증가했습니다. 실험 1회당 약 30분 이상 소요됩니다.
