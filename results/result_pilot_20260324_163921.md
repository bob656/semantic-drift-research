## 평가 방식

### 평가 구조

```
평가자: qwen2.5-coder:7b (temperature=0.1)
입력:  생성된 코드 + 해당 단계의 수정 요청
출력:  숫자 하나 (0~10)
```

각 수정 단계에서 **해당 단계의 수정 요청만** 을 기준으로 평가합니다. 이전 단계에서 요구한 기능이 망가졌는지는 묻지 않습니다.

### 평가 기준 (프롬프트 내용)

| 기준 | 비중 |
|---|---|
| 요구사항 충족도 (해당 단계 요청만) | 40% |
| 코드 품질 (가독성, 구조, 네이밍) | 30% |
| 오류 방지 (명백한 버그, 문법 오류) | 30% |

### 평가 방식의 한계

의미 드리프트 연구의 핵심은 **"이전 기능이 망가졌는가"** 인데, 현재 평가자는 그것을 전혀 측정하지 않습니다. 수정 4단계 평가 시 수정 1~3의 메서드가 사라져도 점수에 반영되지 않는 구조입니다. 이것이 모든 점수가 8~9점에 머무른 근본 원인 중 하나입니다.

**개선 방향:** 평가 기준을 해당 단계 요청이 아닌 누적 요구사항으로 변경해야 합니다.

```python
# 현재: 해당 단계 요청만 전달
self.evaluator.evaluate(current_code, modification)

# 개선: 초기 과제 + 지금까지의 모든 수정 요청을 합산해서 전달
full_requirements = initial_task + "\n" + "\n".join(modifications_so_far)
self.evaluator.evaluate(current_code, full_requirements)
```

## 문제 분석

### 1. StateDoc 점수가 3번 모두 완전히 동일
3회 실험 모두 `[8.5, 8.5, 9.0, 8.5, 8.5]`로 동일.
**원인:** `temperature=0.1`로 설정되어 LLM이 사실상 결정론적으로 동작. 평가자(CodeEvaluator)도 동일한 조건이므로 같은 코드에 항상 같은 점수를 부여. 3번 실험이 독립적인 샘플이 아님.

### 2. 천장 효과 (Ceiling Effect)
모든 점수가 8.0~9.0에 밀집. 드리프트를 측정하려면 점수 하락이 있어야 하나 두 에이전트 모두 고득점 유지.
**원인:** StudentManager 시나리오가 해당 모델에게 너무 쉬움. 평가 LLM이 관대하게 채점. 또한 평가가 누적 요구사항이 아닌 해당 단계 요청만을 기준으로 하여 드리프트를 포착하지 못함.

### 3. 통계적 유의성 없음
p-value = 0.374 (기준 0.05 미만). 효과 크기(Cohen's d = 0.816)는 크지만 샘플 3개로는 검증 불가.
